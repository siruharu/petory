# 대시보드/반려동물 등록 UX 후속 계획

작성일: 2026-03-24  
기준 문서: `docs/review-dashboard.md`  
목적: 대시보드에서 반려동물 등록으로 이어지는 흐름을 유지하면서, 현재 구현에서 남아 있는 실제 사용 문제인 반려동물 이미지의 실제 파일 선택/업로드 UX 부재, 등록 후 홈 미반영, 몸무게 소수점 입력, 품종 선택 UI 붕괴, 웹 홈 대시보드 스크롤 문제, 다중 반려동물 전환 정렬 정책 미정 문제를 우선 해결하기 위한 계획을 정리한다.  
중요: 이 문서는 설계 문서다. 구현은 포함하지 않는다.

## 0. 전제

현재 상태를 기준으로 확인된 점:

- 홈 → 반려동물 목록/등록 진입 경로는 현재 구현에서 닫혔다.
- 종/성별/중성화 상태 선택형 입력, 품종 검색 UI, 생일 달력, 나이 자동 계산, 웹 스크롤 구조는 이미 1차 반영됐다.
- 따라서 현재 문제는 “선택형 입력이 없는 상태”가 아니라, “구현된 UX가 실제로 안정적으로 동작하지 않는 상태”다.

명시적 가정:

- 기존 단순 상태 분기 구조는 이번 단계에서 유지한다.
- 이번 우선순위는 “반려동물 등록 후 결과가 홈에 즉시 반영되고, 폼 입력이 실제로 끊기지 않게 만드는 것”이다.
- 지원 종은 MVP 기준으로 우선 `dog`, `cat` 중심으로 본다.
- 품종 후보군은 최소한 개/고양이 기준으로 구분된 선택지를 제공한다.
- 나이는 생일이 있으면 자동 계산하고, 수동 나이 입력은 이번 단계에서도 추가하지 않는다.
- 현재 홈 미반영 문제는 홈 재조회 누락, 로컬 상태 미동기화, 또는 백엔드 응답 반영 지점 부재 중 하나로 본다.
- 몸무게 입력은 사용자가 입력 중인 문자열 상태와 저장 직전 숫자 변환을 분리하는 방향이 가장 안전하다고 본다.
- 반려동물 이미지는 URL 입력이 아니라 사용자가 직접 파일을 선택하거나 업로드하는 UX가 더 적절하다고 본다.
- 웹 홈 대시보드의 스크롤 문제는 화면 구조 자체의 문제로 간주하고, 홈 데이터 양과 무관하게 항상 하단까지 접근 가능해야 한다.
- 다중 반려동물 전환 UX는 “현재 선택 항목을 첫 위치로 올릴지” 여부를 명시적으로 정해야 한다.

## 1. 목표

이번 계획의 목표는 아래 다섯 가지다.

1. 반려동물 등록 폼에서 이미지 파일을 직접 선택하거나 업로드할 수 있는 UX를 정의하고 반영한다.
2. 반려동물 등록 성공 후 홈으로 돌아왔을 때 신규 반려동물이 즉시 보이도록 동기화한다.
3. 몸무게(`weightKg`) 입력이 소수점을 포함해 안정적으로 동작하도록 수정한다.
4. 품종 검색/선택 UI를 웹 기준으로 읽기 쉽고 선택 가능한 구조로 다시 정리한다.
5. 홈 대시보드가 웹에서도 항상 스크롤 가능하도록 수정한다.
6. 여러 반려동물 전환 시 현재 선택 항목의 정렬 정책을 명시한다.
7. 이미 반영된 선택형 입력과 생일/나이 UX를 깨뜨리지 않고 유지한다.
8. 반려동물 등록 화면이 웹에서도 끝까지 읽히고 조작 가능한 상태를 유지한다.

## 2. 핵심 문제 정의

현재 반려동물 등록 UX의 문제는 아래 다섯 가지다.

1. 반려동물 등록 성공 후 홈으로 복귀해도 “등록된 반려동물이 없어요” 상태가 보일 수 있다.
2. 몸무게 입력은 소수점 입력 도중 값이 끊기거나 반영되지 않는 문제가 있다.
3. 품종 검색/선택 UI는 기능은 있지만 웹에서 리스트가 시각적으로 깨져 보인다.
4. 반려동물 등록 폼의 이미지는 현재 URL 입력 기반이라 일반 사용자에게 자연스럽지 않다.
5. 홈 대시보드는 웹에서 스크롤되지 않아 하단 섹션 확인이 어렵다.
6. 여러 반려동물을 등록한 뒤 전환할 때 현재 선택 항목을 목록 첫 위치로 올릴지 정책이 없다.
7. 대표 품종 리스트 기반이라 일부 품종은 여전히 후보군에 없을 수 있다.
8. 생일 미상 사용자에 대한 대체 UX는 여전히 단순하다.

즉, 현재 문제는 “입력 필드 부재”가 아니라 “입력 안정성과 등록 결과 반영”이다.

## 3. 우선순위

### 3.1 1순위: 반려동물 이미지의 실제 파일 선택/업로드 UX

핵심 문제:

- 현재 이미지 등록은 URL 입력 기반이라 일반 사용자에게 부자연스럽다.

처리 방향:

- 웹 기준으로 파일 선택 기반 이미지 입력 UX를 우선 설계한다.
- 최소 범위는 아래처럼 본다.
  - 파일 선택 버튼
  - 선택된 이미지 preview
  - 업로드 후 `photoUrl` 또는 업로드 결과값을 payload에 반영
- 업로드 인프라가 아직 없으면 로컬 preview + 업로드 placeholder 구조까지 어디를 MVP로 볼지 명확히 정한다.

완료 기준:

- 사용자가 이미지 URL을 직접 입력하지 않고도 반려동물 이미지를 선택할 수 있다.
- 선택된 이미지가 폼과 홈 카드 흐름에서 자연스럽게 이어진다.

### 3.2 2순위: 등록 성공 후 홈 반영 동기화

핵심 문제:

- 반려동물 등록이 성공해도 홈 화면에 즉시 반영되지 않아 사용자가 저장 결과를 신뢰하기 어렵다.

처리 방향:

- 등록 성공 시점에 홈이 다시 그려질 수 있는 최소 동기화 경로를 만든다.
- 구현 후보는 아래 둘 중 하나다.
  - 홈 복귀 시 강제 재조회
  - 등록 성공 payload를 루트 상태에 반영한 뒤 홈에서 사용
- 선택 상태까지 포함할지 여부를 명확히 한다.

완료 기준:

- 등록 후 홈에서 “등록된 반려동물이 없어요” 상태가 더 이상 남지 않는다.
- 최소한 새 반려동물 목록 반영은 즉시 보장된다.

### 3.3 3순위: 몸무게 소수점 입력 안정화

핵심 문제:

- 몸무게 입력값을 숫자로 즉시 변환하면 `3.`, `3.5` 같은 입력 중간 상태를 잃기 쉽다.

처리 방향:

- 폼 상태에서는 몸무게를 문자열 draft로 유지한다.
- 제출 직전 또는 validation 시점에만 숫자로 변환한다.
- 웹 입력 기준으로 `.` 입력과 소수 둘째 자리 수준까지 자연스럽게 입력되는지 본다.

완료 기준:

- 몸무게 필드에서 소수점 입력이 끊기지 않는다.
- 저장 payload는 여전히 숫자 또는 null 형태를 유지한다.

### 3.4 4순위: 품종 검색/선택 UI 재정리

핵심 문제:

- 품종 검색 컴포넌트는 기능은 있으나 웹에서 검색 input, 목록, 선택 상태의 위계가 약해 리스트가 깨져 보인다.

처리 방향:

- `search-select-field`의 박스 구조, 간격, border, 옵션 row, 최대 높이를 정리한다.
- 검색 input과 옵션 목록을 시각적으로 분리한다.
- 선택된 품종이 현재 값으로 분명히 보이도록 헤더 표현을 보강한다.

완료 기준:

- 웹에서 품종 선택 UI가 깨져 보이지 않는다.
- 사용자가 “현재 선택값 / 검색 / 후보 목록”을 즉시 구분할 수 있다.

### 3.5 5순위: 홈 웹 스크롤 보강

핵심 문제:

- 홈 대시보드는 웹에서 하단 콘텐츠까지 스크롤되지 않아 정보 확인이 어렵다.

처리 방향:

- `home-screen.tsx` 또는 상위 container에서 홈을 스크롤 가능한 구조로 감싼다.
- 홈 카드, 일정, 최근 기록이 길어져도 하단까지 접근 가능해야 한다.
- 웹에서 상단 segmented header와 본문이 자연스럽게 이어지도록 한다.

완료 기준:

- 웹 브라우저에서 홈 하단 섹션까지 스크롤로 접근 가능하다.

### 3.6 6순위: 다중 반려동물 전환 정렬 정책 정의

핵심 문제:

- 여러 반려동물 전환 시 현재 선택 항목을 첫 위치로 올리는 것이 자연스러운지 제품 기준이 아직 없다.

처리 방향:

- 아래 후보 중 하나를 선택한다.
  - 선택 상태만 바꾸고 원래 등록 순서를 유지한다
  - 최근 선택한 반려동물을 첫 위치로 올린다
  - 별도 pin/favorite 정책이 생기기 전까지 현재 선택 강조만 유지한다
- 홈의 반려동물 switcher와 목록 화면이 같은 정렬 정책을 따르도록 한다.

완료 기준:

- 다중 반려동물 전환 시 목록 정렬과 선택 강조 기준이 문서와 구현에서 일치한다.

### 3.7 7순위: 기존 선택형 입력과 생일/나이 UX 유지

핵심 문제:

- 앞선 수정 과정에서 이미 반영된 종/성별/중성화 선택, 생일 달력, 나이 자동 계산이 회귀하면 안 된다.

처리 방향:

- 폼 상태 구조 변경 시 `species`, `breed`, `sex`, `neuteredStatus`, `birthDate`, `ageText`의 현재 동작을 유지한다.
- `species` 변경 시 `breed` reset, `birthDate` 변경 시 `ageText` 재계산 규칙을 유지한다.

완료 기준:

- 기존 선택형 입력과 생일/나이 연동이 깨지지 않는다.

### 3.8 8순위: 등록 폼 웹 스크롤과 하단 접근성 유지

핵심 문제:

- 폼 UI를 손보는 과정에서 웹 스크롤과 하단 제출 영역 접근성이 다시 깨질 수 있다.

처리 방향:

- `ScreenContainer`의 scrollable 구조를 유지한다.
- 품종 검색 리스트가 길어져도 전체 화면 스크롤과 내부 리스트 스크롤이 충돌하지 않도록 본다.

완료 기준:

- 웹에서 폼 하단과 제출 버튼까지 계속 접근 가능하다.

## 5. 단계별 실행 계획

### Step 1. 이미지 파일 선택/업로드 UX 기준 정리

목적:

- URL 입력이 아닌 실제 사용자 기준의 이미지 등록 경로를 정의한다.

세부 항목:

- 웹 파일 선택 방식 검토
- preview와 payload 연결 방식 정의
- 업로드 인프라 미정 시 placeholder 범위 결정

### Step 2. 등록 후 홈 동기화 경로 정리

목적:

- 반려동물 등록 성공 이벤트가 홈 데이터와 어떻게 연결될지 닫는다.

세부 항목:

- 등록 성공 시 상위 상태로 전달할 최소 데이터 정의
- 홈 복귀 시점 재조회 여부 결정
- empty state 제거 기준 정의

### Step 3. 몸무게 입력 상태 분리

목적:

- 소수점 입력이 깨지지 않도록 폼 상태와 저장 payload를 분리한다.

세부 항목:

- weight draft를 문자열로 유지하는 구조 검토
- submit 직전 number 변환 기준 정의
- invalid decimal 입력 처리 기준 정의

### Step 4. 품종 검색/선택 UI 레이아웃 정리

목적:

- 웹에서 품종 리스트가 깨져 보이지 않도록 시각 구조를 정리한다.

세부 항목:

- search-select header 정리
- 검색 input과 옵션 목록 spacing/border 정리
- 옵션 row 가독성 정리

### Step 5. 홈 웹 스크롤 정리

목적:

- 홈 대시보드에서 하단 카드와 리스트가 웹에서도 항상 보이게 한다.

세부 항목:

- 홈 screen scroll container 적용
- 웹에서 header/body 높이 충돌 여부 점검
- 긴 콘텐츠 기준 스크롤 확인

### Step 6. 반려동물 전환 정렬 정책 정리

목적:

- 여러 반려동물 전환 시 현재 선택 항목의 위치 정책을 명시한다.

세부 항목:

- 선택 항목 first 정렬 여부 결정
- switcher/list/home에 동일 정책 적용 범위 정의
- 선택 강조와 정렬을 분리할지 여부 결정

### Step 7. 기존 선택형 입력/생일 UX 회귀 점검

목적:

- 새 수정으로 기존 드롭다운/달력/나이 자동 계산이 깨지지 않게 한다.

세부 항목:

- species 변경 시 breed reset 확인
- birthDate 변경 시 ageText 재계산 확인
- sex/neuteredStatus 선택 상태 유지 확인

### Step 8. 웹 스크롤과 최종 흐름 점검

목적:

- 실제 등록 흐름 기준으로 “입력 → 저장 → 홈 반영”이 닫히는지 확인한다.

세부 항목:

- 품종 검색/선택 UI 확인
- 몸무게 소수점 입력 확인
- 이미지 파일 선택/표시 확인
- 등록 후 홈 반영 확인
- 홈 웹 스크롤 확인
- 다중 반려동물 전환 정렬 확인
- 등록 폼 웹 스크롤 확인

## 6. 트레이드오프

### 6.1 등록 후 홈을 재조회로만 동기화하는 경우

장점:

- 구현이 단순하다.
- 서버가 실제 저장 결과를 source of truth로 유지할 수 있다.

단점:

- 네트워크 지연이 있으면 홈 반영이 느릴 수 있다.
- 백엔드 응답이 아직 빈 데이터면 문제를 가린 채 남길 수 있다.

현재 판단:

- 우선은 재조회 기반이 가장 안전하지만, 필요하면 등록 응답을 로컬 상태에도 반영해야 한다.

### 6.2 몸무게를 즉시 number로 변환하는 경우

장점:

- 상태 타입이 단순하다.

단점:

- 소수점 입력 중간 상태를 잃기 쉽다.
- 웹 입력 UX가 깨지기 쉽다.

현재 판단:

- 폼 상태는 문자열, 제출 payload는 number로 분리하는 편이 적절하다.

### 6.3 품종 드롭다운만 사용하는 경우

장점:

- 구현이 단순하다.
- 모바일/웹 모두 빠르게 안정화할 수 있다.

단점:

- 후보군이 많아지면 탐색성이 떨어진다.

### 6.2 검색형 품종 선택을 도입하는 경우

장점:

- 품종 수가 많아도 찾기 쉽다.

단점:

- 입력 컴포넌트 복잡도가 올라간다.
- MVP 범위를 넘기 쉬워진다.

현재 판단:

- 초기에는 드롭다운 기반 또는 매우 단순한 검색형 선택으로 제한하는 편이 적절하다.

### 6.4 이미지 URL 입력만 유지하는 경우

장점:

- 구현이 가장 단순하다.
- 현재 계약을 거의 건드리지 않는다.

단점:

- 일반 사용자 UX가 부자연스럽다.
- 실제 프로필 이미지 등록률이 낮을 가능성이 높다.

현재 판단:

- 더 이상 기본안으로 보기 어렵다. 최소한 파일 선택 UX는 우선 설계해야 한다.

### 6.5 이미지 업로드를 실제 파일 업로드로 바로 가는 경우

장점:

- 사용자 경험이 직관적이다.
- 홈 카드에 바로 프로필 이미지를 보여주기 좋다.

단점:

- 저장소/업로드 API/preview까지 함께 정리해야 해 범위가 커진다.

현재 판단:

- 최종 목표는 이 방향이지만, 업로드 인프라 범위를 MVP에서 어디까지 볼지 명확히 잘라야 한다.

### 6.6 선택한 반려동물을 첫 위치로 올리는 경우

장점:

- 사용자가 방금 선택한 반려동물을 다시 찾기 쉽다.
- 다중 반려동물 환경에서 현재 컨텍스트가 더 분명해질 수 있다.

단점:

- 등록 순서나 고정 순서를 기대하는 사용자는 목록이 흔들린다고 느낄 수 있다.
- 화면마다 정렬 기준이 다르면 더 혼란스럽다.

현재 판단:

- 단순히 “첫 위치로 올리는 것”이 자연스럽다고 단정하면 안 된다. 선택 강조만 유지하고 정렬은 고정하는 방식과 비교 검토가 필요하다.

### 6.7 나이 자동 계산만 사용하는 경우

장점:

- 데이터 일관성이 높아진다.

단점:

- 생일을 모르는 사용자는 나이를 직접 적고 싶어 할 수 있다.

현재 판단:

- MVP에서는 생일이 있으면 자동 계산, 생일이 없으면 나이 표시 없음 또는 안내 문구 처리로 단순화하는 편이 적절하다.

## 7. 리스크

### 높음

- 실제 파일 선택/업로드 UX가 없으면 이미지 등록 기능이 사실상 반쪽짜리로 남는다.
- 등록 후 홈 반영이 계속 실패하면 사용자가 저장 성공 자체를 신뢰하지 못한다.
- 홈 웹 스크롤이 계속 막히면 대시보드 핵심 정보 접근성이 떨어진다.

### 중간

- 몸무게 소수점 입력 불안정이 계속되면 기본 정보 입력 완성도가 크게 떨어진다.
- 품종 검색 UI가 계속 깨져 보이면 웹 프리뷰 품질이 낮아 보인다.
- 다중 반려동물 전환 정렬 정책이 없으면 전환 UX 기대치가 흔들릴 수 있다.
- 품종 후보군 범위를 어떻게 자를지에 따라 입력 UX 만족도가 달라질 수 있다.
- 나이 자동 계산 정책이 생일 미상 사용자에게 다소 불편할 수 있다.

### 낮음

- 수동 라우팅 구조 자체는 이번 수정의 핵심 리스크는 아니다.

## 8. 완료 기준

이 계획이 완료됐다고 볼 기준은 아래와 같다.

- 반려동물 등록 폼에서 사용자가 직접 이미지 파일을 선택하거나 업로드할 수 있다.
- 반려동물 등록 후 홈으로 복귀했을 때 신규 반려동물이 즉시 보인다.
- 몸무게 필드에서 소수점 입력이 자연스럽게 동작한다.
- 품종 검색/선택 UI가 웹에서 깨져 보이지 않는다.
- 홈 화면이 웹에서 하단까지 스크롤된다.
- 다중 반려동물 전환 시 정렬/선택 정책이 명시적으로 정해진다.
- 종, 성별, 중성화 상태 선택과 생일/나이 자동 계산이 기존대로 유지된다.
- 웹에서 반려동물 등록 화면 하단까지 스크롤로 접근 가능하다.
