# UI/UX 연구

작성일: 2026-03-24  
대상: `app/` 기반 반려동물 건강/생활 관리 앱  
목적: 2026년 기준 최신 모바일 앱 UI/UX 패턴을 조사하고, 현재 Petory 앱에 적용 가능한 화면/컴포넌트/피그마 파일 구조 기준을 정의한다.  
중요: 이 문서는 설계와 레퍼런스 연구 문서다. 구현은 포함하지 않는다.

## 0. 전제와 해석 범위

- 기준 플랫폼은 모바일 앱이다.
- 다만 현재 검증 환경은 Expo web도 함께 고려하므로, 정보 밀도와 인터랙션은 모바일 퍼스트이되 웹에서도 구조가 무너지지 않아야 한다.
- 비교 레퍼런스는 2026년 시점의 Apple iOS 26, Figma UI kit, Toss의 UX 원칙, 당근의 제품 방향성/브랜드 특성, Google/Android의 최신 모바일 패턴을 함께 본다.
- Toss와 당근은 공개된 공식 아티클, 제품/채용/보도자료, 앱 스토어 설명을 바탕으로 참고한다.
- 당근 관련 일부 해석은 공개 소스상 “디자인 원칙/SEED/브랜드 방향”을 근거로 한 추론이다.

## 1. 현재 앱 UI 상태 진단

현재 `app/src/screens/*`를 보면 UI는 기능 스켈레톤 수준이다.

관찰:

- 화면 대부분이 `Text`, `Button`, `TextInput`, 단순 `View` 조합으로만 구성되어 있다.
- 타입과 API 흐름은 어느 정도 연결됐지만, 화면 정보 구조와 시각적 계층은 거의 정의되지 않았다.
- `home-screen.tsx`는 “선택된 반려동물 / 오늘 일정 / 미뤄진 일정 / 최근 기록”이라는 올바른 정보 구조를 이미 갖고 있으나, 이를 카드/배지/우선순위 묶음으로 표현하지 못하고 있다.
- `login-screen.tsx`와 `signup-screen.tsx`는 실제 동작은 연결됐지만, 입력 보조, CTA 우선순위, 상태 메시지 톤, 오류 복구 흐름이 매우 원시적이다.
- `pet-list`, `schedule-list`, `record-list`는 목록은 조회하지만, 2026년형 앱에서 기대되는 “요약-행동-상태-다음 액션” 구조가 없다.

결론:

- 현재 코드베이스는 UX 흐름의 골격은 맞다.
- 그러나 시각 체계, 상호작용 체계, 정보 밀도 조절, 상태 설계, 감정 톤이 비어 있다.
- 따라서 이번 연구의 핵심은 “기능을 더 추가”하는 것이 아니라 “같은 기능을 최신 앱 수준의 경험으로 재구성”하는 데 있다.

## 2. 2026년 모바일 앱 UI/UX 핵심 트렌드

### 2.1 트렌드 1: 콘텐츠 우선, 크롬 최소화

2026년 기준 상위 앱들은 상단/하단 바를 무겁게 두지 않는다.  
시스템 크롬은 더 가벼워지고, 실제 콘텐츠가 전면에 나온다.

Petory 적용:

- 홈 화면에서 가장 먼저 보여야 하는 것은 “앱 소개”가 아니라 “오늘 해야 할 돌봄”이다.
- 상단에는 큰 브랜드 헤더보다 `오늘`, `모카`, `2개 미완료` 같은 현재 컨텍스트가 우선이다.
- 바텀탭이나 상단바는 반투명 또는 낮은 대비로 두고, 카드/콘텐츠가 시선을 받게 해야 한다.

### 2.2 트렌드 2: 정보는 한 화면에서 빠르게 스캔 가능해야 한다

Toss 계열 UX는 오래전부터 “한 페이지 한 핵심 메시지”를 밀고 있었고, 2026년에도 여전히 유효하다.  
다만 최근 앱들은 한 화면에 여러 블록을 두더라도, 각 블록마다 “한 블록 한 목적”이 명확하다.

Petory 적용:

- 홈 카드 하나당 목적은 하나여야 한다.
- 예:
  - `오늘 급여 1건`
  - `약 복용 2시간 후`
  - `체중 기록 5일째 없음`
- 한 카드에서 일정 상세, 기록 그래프, 병원 메모를 모두 보여주면 실패한다.

### 2.3 트렌드 3: 바텀시트는 “서브 화면”이 아니라 “빠른 결정 레이어”다

2025~2026 패턴에서 바텀시트는 매우 강한 기본 패턴이다.

적합한 사용처:

- 반려동물 전환
- 일정 타입 선택
- 기록 타입 선택
- 완료 액션 확인
- 필터 선택

부적합한 사용처:

- 긴 입력 폼 전체
- 복잡한 멀티스텝 생성 흐름
- 인증/온보딩 메인 플로우

Petory 적용:

- `PetSwitcherSheet`는 유지하되, 단순 리스트가 아니라 “현재 선택 / 다른 반려동물 / 새 반려동물 추가” 3구역 구조가 맞다.
- `기록 추가`는 먼저 타입 선택 바텀시트, 그 다음 전체화면 폼으로 가는 2단계가 맞다.

### 2.4 트렌드 4: 모션은 적게, 그러나 문맥적으로 정확해야 한다

2026년 상위 앱들은 인터랙션을 남발하지 않는다.  
대신 다음 3가지에만 집중한다.

- 상태 전환이 이해에 도움될 때
- 사용자의 성공/완료를 감정적으로 강화할 때
- 공간 관계를 설명해야 할 때

Petory 적용:

- 반려동물 전환 시 전체 화면이 번쩍 바뀌는 대신, 상단 헤더와 홈 카드 일부만 부드럽게 교체되어야 한다.
- 일정 완료 시 카드가 즉시 사라지는 것보다, 체크 완료 후 축소/정렬 전환이 더 낫다.
- 로그인 실패는 흔들림 애니메이션보다, 입력 필드 강조와 오류 문구 노출이 낫다.

### 2.5 트렌드 5: AI 감성보다 “실무형 요약 UX”가 강하다

2026년에는 AI를 전면에 내세우는 앱도 많지만, 실제 생산성/생활 앱의 좋은 UX는 “AI처럼 보이는 것”보다 “사용자 대신 요약해주는 것”에 가깝다.

Petory 적용:

- 홈 첫 카드에 “오늘 돌봄 요약”을 둔다.
- 예:
  - `오늘은 급여 1건, 산책 1건, 약 복용 1건이 있어요`
  - `모카의 체중 기록이 8일째 비어 있어요`
- 하지만 챗봇 UI를 전면에 두는 것은 MVP 기준 과하다.

### 2.6 트렌드 6: 감성은 강하지만, 버튼과 텍스트는 더 직접적이다

최신 앱은 배경, 일러스트, 카드 질감은 감성적으로 쓰지만, 버튼과 본문 문구는 오히려 더 직접적이다.

Petory 적용:

- 감성은 사진, 카드 배경, 상태 컬러, 미세한 모션에서 처리한다.
- CTA는 직설적으로 쓴다.
  - `기록 추가`
  - `오늘 일정 완료`
  - `인증 메일 다시 보내기`
- 추상적 CTA는 피한다.
  - `계속하기`
  - `다음`
  - `확인`

### 2.7 트렌드 7: 접근성은 옵션이 아니라 기본 레이아웃 제약이다

2025~2026 기준 공식 가이드는 Dynamic Type, contrast, semantic structure, keyboard/screen reader 친화성을 더 강하게 요구한다.

Petory 적용:

- 11pt 이하 텍스트를 쓰지 않는다.
- 주요 본문은 15~17pt 기준으로 설계한다.
- 색만으로 상태를 구분하지 않는다.
- 기록/일정 카드에는 반드시 텍스트 라벨이 있어야 한다.

## 3. Toss에서 가져와야 할 UX 원칙

공개 소스 기준으로 Toss에서 유효하게 참고할 수 있는 것은 “화려한 스타일”보다 “정보 구조와 문구 원칙”이다.

### 3.1 One thing per one page

이 원칙은 Petory에도 매우 잘 맞는다.

적용:

- 로그인 화면: 로그인만
- 회원가입 화면: 가입만
- 이메일 인증 대기 화면: 인증 안내와 재전송만
- 홈 화면: 오늘 해야 할 것과 최근 상태 확인만
- 일정 생성 화면: 일정 입력만

실패 예:

- 로그인 화면에 회원가입 설명, 제품 소개, 반려동물 이미지, 팁, 서버 상태 메시지를 다 넣는 것

### 3.2 CTA는 키보드/입력 흐름을 방해하지 않아야 한다

Toss 가입 경험에서 중요한 포인트는 입력 흐름과 CTA 위치를 함께 설계했다는 점이다.

Petory 적용:

- 로그인/회원가입 CTA는 하단 고정 또는 키보드 위 고정에 가깝게 설계해야 한다.
- 입력 필드가 늘어날수록 버튼이 아래로 밀리는 구조는 피해야 한다.
- 웹 프리뷰에서도 모바일 폭 기준 sticky submit 영역을 유지하는 것이 좋다.

### 3.3 문구는 사용자를 몰아붙이지 않는다

Toss UX writing 원칙의 핵심은 과장, 불안 자극, 난해한 표현을 줄이는 것이다.

Petory 적용:

- 오류 문구:
  - 좋음: `이메일이나 비밀번호가 맞지 않아요`
  - 나쁨: `로그인에 실패했습니다`
- 일정 비어있음:
  - 좋음: `오늘 예정된 일정이 없어요`
  - 나쁨: `데이터가 없습니다`
- 기록 없음:
  - 좋음: `최근 체중 기록이 없어요. 첫 기록을 남겨보세요`

### 3.4 인터랙션은 비용 대비 의미가 있을 때만

Toss 인터랙션 아티클 관점에서 보면, Petory는 모든 전환에 애니메이션을 넣는 것보다 아래 우선순위가 맞다.

우선순위 높은 모션:

- 일정 완료
- 반려동물 전환
- 성공 토스트/인라인 피드백
- 바텀시트 등장/퇴장

우선순위 낮은 모션:

- 단순 리스트 항목 hover
- 버튼 광택 효과
- 과한 로딩 셰이머

## 4. 당근에서 가져와야 할 UX 원칙

당근은 공개된 디자인 문서가 많지 않지만, 공식 채용/브랜딩/제품 설명에서 일관된 방향이 보인다.

### 4.1 서비스는 기능보다 생활 맥락으로 읽혀야 한다

당근은 기능 중심 설명보다 “동네”, “이웃”, “생활”, “연결” 같은 생활 언어로 제품을 설명한다.

Petory 적용:

- 홈 문구를 의료/관리 시스템처럼 쓰지 않는다.
- 예:
  - `모카가 오늘 챙겨야 할 일정이 2개 있어요`
  - `지난 산책 이후 3일이 지났어요`
- 앱이 병원 EMR처럼 보이면 실패다.

### 4.2 따뜻하지만 가볍지 않은 톤

당근식 UX는 친근하지만 유아적이지 않다.

Petory 적용:

- 메인 컬러는 따뜻해도 된다.
- 하지만 아이콘과 일러스트가 지나치게 귀엽기만 하면 건강 관리 앱의 신뢰가 떨어진다.
- 감성 비중은 30%, 신뢰 비중은 70%가 맞다.

### 4.3 커뮤니티 앱처럼 “사진과 카드”를 다루되, 정보 앱처럼 “구조”를 잃지 않는다

Petory 적용:

- 반려동물 카드에는 사진을 넣을 수 있다.
- 하지만 일정/기록 정보가 사진에 밀리면 안 된다.
- 홈 상단 히어로는 사진 기반, 하단 블록은 구조 기반이 적절하다.

### 4.4 신뢰 단서는 작게 자주 보여준다

당근의 공식 제품 설명은 반복적으로 “검증된 이웃”과 신뢰를 강조한다.

Petory 적용:

- 병원 기록, 약 복용, 체중 같은 민감한 데이터는 “마지막 업데이트 시각”을 자주 보여줘야 한다.
- 예:
  - `마지막 체중 기록 3일 전`
  - `병원 메모 2건`
  - `오늘 오전 9:00 완료`

## 5. Petory에 맞는 2026 UI 방향 제안

### 5.1 핵심 방향

Petory는 아래 조합이 맞다.

- Toss식 정보 구조
- 당근식 생활 톤
- Apple/2026식 가벼운 크롬과 부드러운 재질감
- Material/Android식 명확한 바텀시트, 리스트, 적응형 구조

즉, “금융앱처럼 차갑고 건조한 관리 앱”도 아니고, “반려동물 SNS처럼 감성만 강한 앱”도 아니다.  
정답은 “생활형 건강 관리 대시보드”다.

### 5.2 시각적 톤 제안

#### 추천 톤

- 베이스: warm neutral
- 포인트: soft orange / leaf green / alert coral / calm blue
- 질감: flat 단색보다 얕은 gradient와 반투명 레이어
- 카드: 완전 흰색 박스보다 미세한 tinted surface

#### 피해야 할 톤

- 보라색 SaaS 템플릿 톤
- 새하얀 배경 + 검정 텍스트만 있는 의료 앱 톤
- 지나치게 캐릭터 중심인 펫 커뮤니티 톤
- 진한 그림자 남발

### 5.3 추천 정보 우선순위

#### 홈

1. 선택된 반려동물
2. 오늘 해야 할 일정
3. 놓친 일정
4. 최근 기록
5. 빠른 액션

#### 일정

1. 오늘/다가오는 일정
2. 완료 여부
3. 일정 종류
4. 반복 여부
5. 메모

#### 기록

1. 기록 종류
2. 발생 시각
3. 핵심 값
4. 메모
5. 일정 연계 여부

## 6. 화면별 UI/UX 연구

## 6.1 로그인

목표:

- 빠르게 들어오게 한다.
- 오류 복구가 쉽다.
- 회원가입보다 로그인 행동을 우선시한다.

구조:

- 상단: 앱 이름 또는 짧은 가치 제안
- 중앙: 이메일, 비밀번호
- 하단 고정: `로그인`
- 보조 링크: `이메일 인증이 안 왔어요`

상태:

- 기본
- 입력 중
- 제출 중
- 비밀번호 오류
- 미인증 이메일

엣지 케이스:

- 이메일만 입력
- 비밀번호만 입력
- 서버 다운
- 401/403 혼재
- 이미 로그인된 세션

피그마 포인트:

- 키보드 올라온 상태 프레임을 별도 만든다.
- CTA가 가려지는지 반드시 확인한다.

## 6.2 회원가입 / 이메일 인증 대기

핵심:

- 가입 성공 직후 access token을 주지 않는 현재 서버 설계와 맞물리므로, “이제 무엇을 해야 하는지”가 UX 핵심이다.

구조:

- 가입 폼
- 성공 후 상태 카드
  - `인증 메일을 보냈어요`
  - `메일이 안 오면 다시 보내기`
  - `메일 인증 후 로그인`

엣지 케이스:

- 메일 재전송 연타
- 스팸함 안내 필요
- 잘못된 메일 주소 가입
- 이미 인증된 계정 재전송

## 6.3 홈

핵심:

- Petory의 가장 중요한 화면이다.
- “관리 화면”이 아니라 “오늘의 돌봄 대시보드”처럼 보여야 한다.

추천 섹션:

1. 펫 컨텍스트 헤더
2. 오늘 해야 할 일정 카드
3. 놓친 일정 경고 카드
4. 최근 기록 카드
5. 빠른 액션 바

상세:

- 펫 헤더에는 이름, 종/나이, 작은 아바타, 상태 배지를 둔다.
- 오늘 일정은 개수보다 항목 우선으로 보여준다.
- overdue는 시각적으로 가장 강하게 보여주되 공포스럽지 않게 한다.
- 최근 기록은 최대 3개까지만 노출한다.

엣지 케이스:

- 반려동물 0마리
- 반려동물 1마리
- 반려동물 여러 마리
- 오늘 일정 0개
- overdue만 있는 경우
- 최근 기록이 너무 많은 경우

## 6.4 반려동물 전환

핵심:

- 전환은 잦은 행동이다.
- 그래서 별도 페이지보다 바텀시트가 맞다.

구조:

- 현재 선택된 반려동물
- 다른 반려동물 목록
- `새 반려동물 등록`

UI:

- 각 항목에 사진, 이름, 종, 최근 일정 수 또는 상태 요약
- 현재 선택 항목에는 체크마크와 강조 배경

엣지 케이스:

- 사진 없음
- 이름 중복
- 5마리 이상

## 6.5 일정 목록

핵심:

- 캘린더 앱처럼 만들기보다, “오늘 해야 할 일 리스트” 경험을 우선한다.

구조:

- 상단 필터: 전체 / 오늘 / 다가오는 / 완료
- 카드: 제목, 타입, 시간, 반려동물, 반복 여부, 완료 버튼

UX:

- 완료 버튼은 카드 내부 우측 상단 또는 하단 명확한 영역
- 완료 시 기록 자동 생성이 있으면 미세한 설명 필요

엣지 케이스:

- 같은 시각 일정 다수
- overdue가 오늘보다 많은 경우
- 반복 일정이 다음 일정으로 이어지는 경우

## 6.6 일정 생성

핵심:

- 종류 선택이 제일 중요하다.
- 모든 필드를 처음부터 한 화면에 노출하지 않는다.

추천 순서:

1. 반려동물
2. 일정 종류
3. 제목
4. 일시
5. 반복
6. 메모

패턴:

- 일정 종류는 segmented control보다 선택 카드 또는 바텀시트가 낫다.
- 날짜/시간은 플랫폼 기본 picker 느낌을 따른다.

## 6.7 기록 목록

핵심:

- 기록은 타임라인이 맞다.
- 단, 값의 중요도에 따라 카드 높이를 다르게 한다.

구조:

- 상단 필터: 전체 / 급여 / 산책 / 체중 / 병원 메모
- 카드 2종:
  - compact row
  - expanded note card

예:

- 체중 기록: 값 중심
- 병원 메모: 메모 중심

## 6.8 기록 생성

핵심:

- 기록 타입에 따라 입력 필드가 달라져야 한다.
- 하지만 MVP에서는 “타입 선택 후 동적 폼” 정도면 충분하다.

권장:

- 1단계: 타입 선택
- 2단계: 타입별 폼

엣지 케이스:

- 체중 값만 있고 단위 없음
- 병원 메모가 너무 김
- 일정 완료 후 자동 생성된 기록 수정

## 6.9 설정

핵심:

- 설정 화면은 메뉴 묶음보다 “계정 / 알림 / 세션” 3구역이 단순하다.

추천 구성:

- 계정: 이메일, 인증 여부
- 알림: 권한 상태, 토큰 등록 상태
- 세션: 로그아웃

## 9. 구체적인 스타일 방향 제안

## 9.1 컬러 시스템

- Primary Orange: 따뜻하지만 과하게 채도 높지 않음
- Soft Cream Background: 메인 배경용
- Neutral Stone Gray: 본문/선/비활성
- Leaf Green: 건강/완료 상태
- Coral Red: overdue/오류
- Mist Blue: 정보 카드 및 병원 메모 계열

주의:

- 오렌지를 메인 브랜드로 쓰더라도 모든 CTA를 오렌지로 칠하면 피로하다.
- 오렌지는 핵심 액션과 선택 상태에만 집중적으로 써야 한다.

## 9.2 타이포그래피

권장 방향:

- 헤드라인은 명확하고 부드러운 산세리프
- 본문은 플랫폼 친화적인 시스템 폰트 우선
- 숫자와 시간은 tabular 느낌이 살아야 한다

사용 원칙:

- 화면당 타이포 계층은 4단계 이내
- 한 카드 안에서 굵기 과용 금지

## 9.3 아이콘

권장:

- 선형 아이콘 + 필요 시 채움 강조
- 의료 아이콘보다 생활 아이콘 우선

예:

- 급여: bowl
- 산책: leash / paw
- 체중: scale
- 병원: cross + note 조합

## 10. 문제 가능성

## 10.1 감성 과잉

반려동물 서비스라고 해서 지나치게 귀엽게 가면 일정/기록 화면의 신뢰감이 무너진다.

## 10.2 건강 관리 앱처럼 너무 무거워짐

반대로 의료 서비스처럼 차갑게 가면 생활 루틴 앱의 진입성이 떨어진다.

## 10.3 홈 화면 과밀

홈에 모든 것을 넣고 싶어질 가능성이 높다.  
하지만 MVP 기준 홈은 `오늘 / 놓침 / 최근 / 빠른 행동`만 남기는 것이 좋다.

## 10.4 바텀시트 남용

시트는 강력하지만, 폼 전체를 시트에 넣기 시작하면 사용성이 급격히 나빠진다.

## 10.5 웹 프리뷰와 모바일 실기기 차이

웹에서는 hover, 넓은 폭, 키보드 부재 때문에 좋아 보이는 레이아웃이 모바일 실기기에서 망가질 수 있다.

따라서 피그마도 반드시:

- iPhone compact width 기준
- keyboard up 상태
- long content 상태

를 함께 만들어야 한다.

## 11. Petory에 가장 적합한 최종 UX 방향

한 줄 정의:

`돌봄 루틴을 빠르게 확인하고 바로 행동하게 만드는 생활형 반려동물 관리 앱`

핵심 경험:

- 홈은 요약
- 일정은 실행
- 기록은 회고
- 인증과 설정은 방해 최소화

디자인 원칙 6개:

1. 한 블록에는 한 목적만 둔다.
2. 오늘 해야 할 행동이 가장 먼저 보여야 한다.
3. 입력과 CTA의 거리감을 줄인다.
4. 바텀시트는 빠른 선택에만 쓴다.
5. 감성보다 신뢰를 우선하되, 차갑게 만들지 않는다.
6. 피그마 파일은 토큰-컴포넌트-패턴-스크린-프로토타입 순으로 분리한다.

## 13. 최종 결론

Petory UI/UX의 정답은 “최신 트렌드를 많이 넣는 것”이 아니다.  
정답은 2026년형 시각 체계를 쓰되, 반려동물 돌봄이라는 생활 맥락에 맞게 다음을 결합하는 것이다.

- Toss식 정보 구조와 명확한 CTA
- 당근식 따뜻한 생활 톤
- Apple식 가벼운 크롬과 소재감
- Android/Material식 명확한 바텀시트와 적응형 구조
- Figma 기준의 토큰화된 설계 파일 구조

이 기준으로 피그마 파일을 만들면, 현재 코드베이스의 로그인/홈/반려동물/일정/기록 화면을 모두 일관된 체계 안에서 재설계할 수 있다.
